A case for goliath
I've been hearing and reading a lot of negative attitudes towards big heavy frameworks like Ember, Angular and even jQuery. It seems like there's a shift towards vanilla JS in the wider javascript community, using or deriving from node to simplify things (see Duo or Browserify for examples of what I mean).
There are even those out there that are vehemently against using anything other than vanilla, claiming that developers should just man up and take charge of maintaining browser compatibility, as it is "not that hard". Apparently, most people use modern browsers; please tell that to the ~12% of my current user base still using non-HTML5 compliant browsers.
That's not to say that they don't have a point either. ES5 shims are pretty solid, easy to find and the javascript community is very open source. There are plenty of package managers out there with massive ecosystems of web components using almost vanilla that work even on IE6 with no shims at all. But I have grown to believe that there is always the right tool for the job.
This rings very true for incredibly large, complex web applications & SPAs. You're insane if you attempt these things without at least using jQuery, and you're setting up the maintainers of this behemoth, spaghetti code base to fail. Of course, you could very well do the same using frameworks! but doing so is a much harder task to accomplish.
Some might spout crap about the performance of these frameworks. "ZOMG look I saved 20ms of cpu time with only 300 extras lines of hand-rolled code". They do have a point; the performance is generally much, much worse. But we live in an age where our smart phones are nearly as powerful as desktops, some more so. The cpu & memory usage that a language and/or framework takes up is growing more and more irrelevant in comparison to the resources available. It's time to move on, start pumping out awesome software in minimal time frames because the web is fast changing - it might not be long until the good ol' html/js/css combo becomes obsolete, at least for BAU development.
Like I said, there is always the right tool for the job. Don't make it hard on yourself by omitting jQuery so that you can boast that your app has maximum performance. By the same token, don't pick up Angular just so that you can bind click events in markup with ease. Use your development experience and savviness to determine what to use and don't be afraid to refactor or even rewrite if it is fitting. You will not be judged by the tools you use but by the code you've written. And you can write bad code in any context.
